# 听书智能客服系统项目实战需求说明

## 1. 项目背景与目标

### 1.1 项目背景

在线听书平台面向用户、内容创作者和运营团队提供有声书、广播剧、儿童故事、知识课程、播客等音频内容服务，覆盖内容搜索、订阅购买、在线播放、下载缓存、会员服务、订单支付、售后处理等完整业务链路。随着平台用户规模不断扩大，用户在内容咨询、订单查询、播放问题、会员服务、售后处理等方面的服务请求持续增长，需要构建一套智能客服系统来承载这些高频服务场景。

### 1.2 项目目标

构建一个面向在线听书行业的智能客服系统，支持以下核心目标：

- 自动识别用户咨询意图，提供有声书咨询、会员咨询、订单查询、播放记录查询等问答能力
- 支持退款申请、工单提交等需要多轮引导的任务型对话
- 在多轮对话中持续收集和利用上下文信息（用户身份、订单号、书籍名称、会员信息等）
- 所有对话状态可持久化，支持会话恢复和历史回溯

------

# 2. 用户角色

## 2.1 用户

核心服务对象。通过智能客服咨询有声书、知识课程、会员权益等内容，查询订单状态、播放记录，申请退款或提交投诉。

### 2.2 客服人员

承接系统无法自动处理的问题，负责人工兜底、复杂售后处理和工单跟进。

### 2.3 运营人员

维护知识库内容、FAQ 条目、内容信息及服务流程配置，保障客服系统稳定运行。

------

# 3. 功能需求

## 3.1 信息检索

系统应能从业务数据和知识库中检索相关信息，并以自然语言组织回复，支撑内容咨询、订单查询、播放记录查询、会员咨询、平台规则咨询等场景。

支持：

- 实时业务数据检索：查询有声书、广播剧、知识课程、播客、会员、订单、支付、退款、播放记录等动态数据
- FAQ 检索：匹配会员权益、退款规则、下载规则、版权说明等制度性问题
- 知识库检索：检索平台规则、使用指南、常见问题等通用内容

------

## 3.2 任务型对话

系统应通过多轮对话引导用户完成需要收集多项信息才能处理的业务任务。

支持：

- **订单查询**：收集订单号 → 查询并返回订单状态、支付情况、购买内容等信息
- **播放记录查询**：收集书籍名称或时间范围 → 查询最近播放章节、播放进度、播放时间等
- **退款申请**：收集订单号、退款原因、退款类型 → 创建退款申请并返回处理结果
- **工单提交**：收集工单类型（播放异常/投诉/退款/账号问题）、关联订单号、问题描述 → 创建工单并返回工单编号

------

## 3.3 对话管控

系统应具备完整的多轮对话管理能力。

支持：

- 支持槽位收集：主动追问订单号、书籍名称、退款原因等缺失信息
- 支持上下文保持：持续利用已收集的用户信息、订单信息、内容信息
- 支持流程切换：当前任务执行过程中，用户发起新任务时，暂存当前任务并切换
- 支持流程恢复：新任务完成后恢复之前任务继续处理
- 支持流程取消：用户主动取消当前任务时，清理对应状态并确认取消

------

## 3.4 闲聊兜底

当用户输入不属于明确业务任务或信息查询时，系统应提供自然友好的闲聊回复，并适当引导用户回到业务场景。

------

# 4. 典型业务场景

## 4.1 内容咨询

用户咨询有声书、广播剧、知识课程、播客等内容时，系统应结合内容数据返回清晰回复。

内容类型包括：

- 有声书：作者、主播、章节数、评分、简介等
- 广播剧：配音演员、集数、更新时间等
- 知识课程：讲师、课程时长、适合人群等
- 播客：节目介绍、更新时间、主持人等

```
用户：有没有评分比较高的悬疑有声小说？

系统：
返回符合条件的有声书列表、作者、主播、评分及简介，
并引导用户查看详情。

用户：
《三体》有多少集？

系统：
返回总章节数、已更新章节、主播、时长等信息。
```

------

## 4.2 订单查询

用户提供订单号后，系统返回订单状态、支付情况、购买内容等信息。

```
用户：

帮我查一下订单 ORD20240501010

系统：

返回订单状态（待支付、已支付、已退款等）、购买内容、
订单金额、支付时间等信息。
```

------

## 4.3 播放记录查询

用户查询最近的收听记录或播放进度。

```
用户：

我昨天听到哪一章了？

系统：

返回最近播放的有声书、章节名称、
播放进度及播放时间。

用户：

《鬼吹灯》我听到哪里了？

系统：

返回当前播放章节及收听进度。
```

------

## 4.4 退款申请

用户申请退款时，系统应收集订单号和退款原因。

```
用户：

我要退款。

系统：

收集订单号

↓

收集退款原因

↓

确认退款类型

↓

提交退款申请

↓

返回申请结果。
```

------

## 4.5 工单提交

用户遇到播放异常、账号问题或投诉时，系统应收集必要信息并创建工单。

```
用户：

购买的课程一直播放不了。

系统：

确认工单类型

↓

收集订单号

↓

收集问题描述

↓

创建工单

↓

返回工单编号及预计处理时间。
```

------

# 5. 对话交互要求

## 5.1 消息处理

- 支持接收文本消息和业务对象消息（如用户发送订单对象、书籍对象）
- 收到业务对象消息时，应自动提取对象信息并匹配当前任务所需槽位

------

## 5.2 响应模式

- 支持非流式响应（一次性返回完整回复）
- 支持流式响应（通过 SSE 实时返回回复内容）

------

## 5.3 会话管理

- 支持创建新会话
- 支持获取会话历史消息
- 支持获取当前会话状态（当前任务流程、已收集槽位信息）
- 会话状态应持久化存储，支持会话恢复

------

## 5.4 意图澄清

当系统无法准确判断用户意图时，应主动向用户澄清，而不是直接给出错误回复。

澄清场景包括：

- 用户输入同时匹配多个意图
- 用户输入过于模糊
- 当前任务缺少关键信息
- 用户输入与当前任务上下文存在冲突

------

# 6. 非功能需求

## 6.1 可靠性

- 单次对话处理失败不影响整体服务可用性
- 对外部业务 API 调用具备超时和异常处理能力
- 对话状态保存失败时应有降级处理

------

## 6.2 可观测性

- 应能追踪每次对话处理轨迹（意图识别结果、进入流程、执行动作、生成回复）
- 应能记录关键节点执行状态和耗时
- 工单处理过程支持日志记录与审计

------

## 6.3 准确性

- 查询结果应基于实时业务数据，避免返回错误或过时信息
- 任务流程中的槽位收集应准确提取用户输入中的关键信息
- FAQ 和知识库检索应优先返回最匹配内容

------

## 6.4 可扩展性

- 新增任务流程支持配置化扩展，无需修改核心代码
- 新增知识类型支持灵活扩展
- 新增业务 API 调用支持快速集成

------

# 7. 项目交付物

- 对话接口与会话管理能力
- 一套可运行的听书平台业务样本数据与配置
- 前端调试/演示页面
- 项目说明文档与启动说明

------

# 8. 验收标准

1. 能够完成有声书咨询、广播剧咨询、知识课程咨询、订单查询、播放记录查询等基础问答。
2. 能够在多轮对话中持续收集和使用用户身份、订单号、书籍名称、会员信息等槽位数据。
3. 能够完成退款申请、工单提交等完整的多轮任务流程。
4. 能够完成一次完整的"用户提问 → 意图识别 → 业务查询/流程处理 → 生成回复 → 状态更新"闭环演示。
5. 对话状态可持久化，支持会话恢复。

------

# 业务数据

**audio-data**

主要包括以下业务数据：

- 用户（User）
- 有声书（Audiobook）
- 广播剧（Radio Drama）
- 知识课程（Course）
- 播客节目（Podcast）
- 作者（Author）
- 主播（Narrator）
- 章节（Chapter）
- 会员（Membership）
- 订单（Order）
- 支付记录（Payment）
- 退款记录（Refund）
- 播放记录（Playback History）
- 收藏（Favorite）
- 工单（Ticket）

该版本与旅游场景保持了相同的系统能力（信息检索、多轮任务、状态管理、流程切换、会话恢复等），仅将业务实体和典型业务流程替换为听书平台场景，更贴近在线音频内容平台的智能客服应用。
